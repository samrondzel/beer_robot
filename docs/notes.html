Tracking / ROS2

Tracker Node — from YOLO packet to stable target

This resource explains the pipeline inside the tracker node. It starts from a raw YOLO packet on /detections_raw, picks the best beer detection, converts the bounding box into a center point, runs Kalman prediction and update, then converts the filtered center back into a bounding box.

Why it matters: raw detections jump around. The rest of the robot needs a stable target and velocity estimate, not noisy frame-by-frame boxes.

State estimation

Kalman Filter — prediction and correction

This deeper diagram shows the actual math idea: the state vector stores position and velocity [x, y, vx, vy]. Matrix A predicts natural motion, matrix B can represent the effect of robot actions, H maps full state into camera space, and the Kalman gain decides how much to trust the camera versus the prediction.

In BeerBot terms: the camera says where the beer seems to be; the filter combines that with memory of previous movement to estimate where the beer probably is.

ROS2 architecture

ROS2 Nodes and Topics — modular robot software

The robot pipeline is split into small independent nodes. The camera node publishes images, the detection node publishes raw boxes, the tracker node publishes smoothed detections, and the target manager publishes a command-oriented target state.

This is the first step from “script that runs everything” to a robotics system that can grow.

Embedded systems

Raspberry Pi ↔ ESP32 — the brain-body link

Raspberry Pi handles expensive high-level work: camera, AI, ROS2 and decisions. ESP32 handles low-level real-time-ish control: parsing commands, generating PWM, and later reading encoders.

This split is the bridge from AI software to physical motion.
